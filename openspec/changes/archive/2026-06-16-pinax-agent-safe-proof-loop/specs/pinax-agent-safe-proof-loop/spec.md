## ADDED Requirements

### Requirement: Agent-safe proof loop

Pinax SHALL provide a reproducible local proof loop that demonstrates safe agent-readable Markdown vault operation without adding new provider or cloud dependencies.

#### Scenario: Proof loop runs from capture to controlled apply

- **GIVEN** a temporary local Markdown vault created for proof-loop validation
- **WHEN** the proof loop runs capture, index refresh, search, relationship inspection, vault doctor, repair or organize plan, version snapshot and controlled apply
- **THEN** every write SHALL go through Pinax CLI/application services
- **AND** protected structured assets SHALL be created by Pinax commands, not hand-written by agents
- **AND** the final output SHALL include receipts or evidence showing what changed.

#### Scenario: Agent output is bounded

- **WHEN** proof-loop commands are run with `--json` or `--agent`
- **THEN** output SHALL use the shared projection contract
- **AND** output SHALL include bounded facts and next actions suitable for agents
- **AND** output SHALL NOT include full note bodies unless the user explicitly invoked a local content-display command that permits it.

#### Scenario: Proof loop evidence is redacted

- **WHEN** proof-loop integration tests persist stdout, stderr, receipts, events, fixtures or evidence files
- **THEN** those surfaces SHALL NOT contain raw tokens, Authorization headers, cookies, provider payloads, hidden system prompts, private tool arguments, full note bodies or unapproved plaintext paths.

### Requirement: Proof loop integration evidence

Pinax SHALL generate project-owned integration test evidence for the proof loop.

#### Scenario: Integration run writes evidence

- **WHEN** maintainers run `task test:integration` for the proof loop
- **THEN** Pinax SHALL write evidence under `temp/integration-test-runs/<run-id>/`
- **AND** the run directory SHALL contain `summary.json`, `command.txt`, `stdout.log`, `stderr.log`, `env.json` and `artifacts/`
- **AND** `summary.json` SHALL be generated by the project runner with schema `yeisme.integration_test_evidence.v1`.

#### Scenario: Failure still writes evidence

- **WHEN** the proof-loop integration command fails
- **THEN** the runner SHALL preserve the original non-zero exit code
- **AND** it SHALL still write redacted evidence for diagnosis.

### Requirement: Main path documentation

Pinax SHALL present its primary product path as local-first capture, retrieval, diagnosis, planning and safe apply.

#### Scenario: Command map starts with the core path

- **WHEN** a user reads the README or command map
- **THEN** the recommended first path SHALL show Capture, Retrieve, Diagnose, Plan and Apply safely
- **AND** advanced workflows such as briefing, provider expansion, hosted Cloud and native external SDKs SHALL be clearly separated from the proof-loop path.
