# Pinax Project Memory

- 2026-06-11: Clarified Pinax remote architecture: `pinax api serve` is centralized local REST/RPC access to one server-side vault, while Pinax Cloud Sync is intended as distributed local-first multi-device sync where every device keeps its own local vault and the Cloud backend coordinates encrypted blobs, manifest revisions, CAS commits, and conflicts.
- 2026-06-12: Cloud Sync server and direct transports are now wired: server uses `internal/cloudclient.Transport`, S3/file/rclone direct transports share `internal/cloudsync.ObjectStoreTransport`, and `remote_write=true` is valid after durable revision commit. Older docs/spec text that says server or rclone remain guarded/unwired is stale and should be corrected before release messaging.
- 2026-06-12: Fixed review findings under OpenSpec change `openspec/changes/pinax-project-review-fixes/`: Cloud Sync docs/specs now reflect wired server/file/S3/rclone transports; RPC `Pinax.Note.Read` defaults to bounded `display=card`; REST uses Projection-aware status mapping; server commit requests carry revision/blob/device metadata; unknown direct-store conditional capability now uses lock fallback; dead `internal/sync/executor.go` was removed. Verified with focused tests, CodeGraph, and `task check`.
- 2026-06-11: Added `internal/cloudclient` as the first CLI-side Cloud Sync contract slice. It defines Config/Client, health/current revision/blob batch-check/upload/download/revision commit methods, stable redacted `cloudclient.Error`, and `testdata/cloud_contract.json`. Verified with `go test ./internal/cloudclient -count=1` and `task check`.
- 2026-06-11: Expanded Cloud Sync design: Cloud Sync is now defined as a protocol with pluggable transports, not only a remote Pinax Cloud Server. Supported target design includes server transport, S3-compatible direct transport, rclone/OneDrive direct transport, and embedded Go API/local RPC, all sharing one sync engine and `remote_write=true` only after durable revision commit.
- 2026-06-11: Implemented Pinax Cloud Sync CLI-side code slice: `internal/cloudsync` protocol/transport abstractions, object-store direct transport over `remote.BlobStore`, server transport adapter over `internal/cloudclient`, `cloud backend set s3`, `cloud backend set rclone` configuration, direct file/S3 push and pull with encrypted envelopes, sync-state receipts, conflict copy preservation, and local RPC `Pinax.Sync.Push/Pull`. Verified with focused cloud tests and `task check`.
- 2026-06-11: Configured and smoke-tested local MinIO direct S3 sync. Bucket `pinax-cloud-ec-test` was created on endpoint `http://10.10.1.102:9010`; Pinax S3 backend now persists custom endpoint/region/path-style query options, pushed a note with `remote_write=true`, and pulled it into a second vault successfully. Do not store or echo the raw MinIO secret in docs/final output.
- 2026-06-11: Fixed Pinax Cloud Sync UX for configured MinIO/S3 direct vaults: `pinax sync init --vault <vault>` now reuses existing `.pinax/cloud/config.yaml` or legacy `.pinax/cloud/config.json` instead of requiring endpoint/workspace/device again; `--target` completion returns cloud/s3/git; `--vault` completion includes registered vault aliases plus local directories. Verified live with the configured MinIO vault and `task check`.
- 2026-06-11: Fixed Cloud Sync commands being hijacked by remote API mode. `cloud` and `sync` now stay local even when `remote.api_url` config or `PINAX_API_URL` is set, so configured S3 direct sync no longer returns `remote_command_unsupported` before reaching local sync logic. Verified with regression tests, installed `pinax`, and `task check`.

Current vault Cloud Sync config now lives at `.pinax/cloud/config.yaml` with `secret_ref=profile://ec`; S3 list verification succeeded with 4 objects under the prefix. Never echo the raw profile secret.

New writes use `.pinax/cloud/config.yaml`; S3 direct config is stored as structured `s3.bucket`, `s3.prefix`, `s3.endpoint`, `s3.region`, `s3.profile`, and `s3.path_style` fields instead of a URL-escaped endpoint string. Legacy `.pinax/cloud/config.json` is read-compatible only and removed on new cloud backend writes. Current vault is migrated to YAML using profile `ec`.
