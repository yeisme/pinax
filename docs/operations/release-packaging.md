# Pinax Release Packaging

Pinax releases are produced with GoReleaser from `vX.Y.Z` tags. The same configuration builds GitHub Release archives, `checksums.txt`, source archives, archive SBOMs, direct Linux package assets, a Homebrew formula, and a Scoop manifest. Snapshot runs are rehearsal-only and must not publish package-manager metadata.

## Release channels

| Channel | Artifact or command | Notes |
| --- | --- | --- |
| Go module install | `go install github.com/yeisme/pinax/cmd/pinax@latest` | For Go users. Use a concrete module version when reproducing a release. |
| GitHub Release archives | `pinax_<version>_<os>_<arch>.tar.gz` or `.zip` | Primary universal install path. Verify with `checksums.txt` before running the binary. |
| Source archive | `pinax_<version>_source.tar.gz` | Generated by GoReleaser for release provenance. |
| SBOM | `*.spdx.json` | Generated for archive artifacts. |
| Homebrew | `yeisme/homebrew-tap/Formula/pinax.rb` | Published only from tagged releases with a publisher token. |
| Scoop | `yeisme/scoop-bucket/bucket/pinax.json` | Published only from tagged releases with a publisher token. |
| Linux packages | `.deb`, `.rpm`, `.apk` | Direct GitHub Release assets generated by nFPM. They are not APT, YUM, DNF or Alpine repositories. |
| Chocolatey | Deferred | Do not publish until package metadata, `CHOCOLATEY_API_KEY`, moderation readiness, and Windows install smoke are approved in a later OpenSpec change. |

## Credentials and permissions

The release workflow uses the repository `GITHUB_TOKEN` only for the Pinax GitHub Release in the current repository. Cross-repository package-manager metadata updates use a separate `PUBLISHER_TOKEN` value.

Preferred setup:

1. Create a GitHub App installed only on `yeisme/homebrew-tap` and `yeisme/scoop-bucket`.
2. Grant the app `contents: write` and no broader permissions.
3. Store the app id in the protected `release` environment variable `RELEASE_APP_ID`.
4. Store the private key in the protected `release` environment secret `RELEASE_APP_PRIVATE_KEY`.
5. Let the workflow mint a short-lived token with `actions/create-github-app-token@v3` and pass it to GoReleaser as `PUBLISHER_TOKEN`.

The `release` environment should disable admin bypass and allow deployments only from `v*` tags. Required reviewers are preferred when the repository plan supports them; if GitHub returns a billing-plan error for reviewer protection, keep the tag policy and use the GitHub App token path above.

A fine-grained PAT named `PUBLISHER_TOKEN` is allowed only as an operator fallback and must be restricted to the same tap and bucket repositories. Local validation and snapshot runs must not require either credential.

## Tag flow

Pinax uses standard semantic version tags in its independent repository:

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

Use standalone `vX.Y.Z` tags for Pinax releases so GoReleaser, Go modules, and GitHub Releases agree on the same semantic version.

## Local validation

Before pushing a release tag from `cli/pinax`, run:

```bash
task check
task release:check
task release:package:validate
```

`task release:package:validate` is credential-free. It validates the GoReleaser config, runs a snapshot release with `--skip=publish`, verifies generated checksums when `sha256sum` is available, extracts a generated archive, runs `pinax version` and `pinax --help`, checks for an archive SBOM, and inspects `.deb`, `.rpm`, and `.apk` package metadata when the host has the relevant tools. Missing host inspection tools are reported as skipped.

The target may write `dist/` artifacts, but it must not write a user vault, provider credentials, package-manager repositories, or public release channels.

## Post-release checklist

After a tagged release publishes, verify the public artifacts without provider credentials or user vaults:

1. Download `checksums.txt` and one archive from the GitHub Release.
2. Verify the archive checksum.
3. Extract the archive and run:

   ```bash
   pinax version
   pinax --help
   ```

4. Confirm the release includes a source archive and `*.spdx.json` SBOM artifacts.
5. On macOS with Homebrew available, install the published formula and smoke the binary:

   ```bash
   brew install yeisme/tap/pinax
   pinax version
   pinax --help
   ```

6. On Windows with Scoop available, install from the Yeisme bucket and smoke the binary:

   ```powershell
   scoop bucket add yeisme https://github.com/yeisme/scoop-bucket
   scoop install pinax
   pinax version
   pinax --help
   ```

7. On Linux runners where package tooling is available, install or inspect each direct release asset:
   - `.deb`: `dpkg-deb --field <asset.deb> Package Version Architecture Description`, then install with `apt` or `dpkg` in an isolated runner when allowed.
   - `.rpm`: `rpm -qip <asset.rpm>`, then install with `dnf`/`rpm` in an isolated runner when allowed.
   - `.apk`: inspect with `tar -tzf <asset.apk>`, then install in an Alpine runner when allowed.
8. If signing or attestations are added in a later change, verify them in this checklist before announcing the release.

## Rollback

Prefer forward fixes over rewriting public release history.

1. If the workflow fails before assets are public, stop the run, fix the release configuration, and rerun from the same tag only if no public package index was updated.
2. If a public release is broken, publish a new patch tag such as `vX.Y.(Z+1)`.
3. If Homebrew or Scoop metadata points to a bad artifact, revert the generated tap or bucket commit and publish a corrected patch release.
4. If a release asset is unsafe, remove that asset from the GitHub Release and document the replacement version in release notes.

Do not reuse a public `vX.Y.Z` tag for different bits after users or package indexes may have cached it.
